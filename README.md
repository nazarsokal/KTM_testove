Flight Telemetry Analyzer (KTM Testove)
Цей проєкт призначений для аналізу, обробки та 3D-візуалізації польотної телеметрії з MAVLink логів. Він поєднує бекенд на платформі .NET 8, скрипти обробки даних на Python та сучасний фронтенд на React з використанням 3D-графіки. Для розгортання проєкту на вашому комп'ютері повинні бути попередньо встановлені Node.js, Python третьої версії та .NET 8 SDK.

Встановлення бекенду та .NET 8
Робота серверної частини вимагає наявності .NET 8. Якщо у вас ще не встановлено відповідний SDK, ви можете завантажити його з офіційного сайту Microsoft. Також для користувачів Windows доступне швидке встановлення через термінал (CMD або PowerShell) за допомогою пакетного менеджера winget. Для цього достатньо виконати наведену нижче команду.
winget install Microsoft.DotNet.SDK.8

Після завершення процесу можна перевірити успішність встановлення та поточну версію платформи командою dotnet --version.

Налаштування Python та залежностей
Для парсингу MAVLink-файлів та математичних розрахунків кінематики польоту проєкт використовує мову Python. Переконайтеся, що інтерпретатор встановлено, після чого завантажте спеціалізовані бібліотеки для роботи з масивами даних. Це робиться через стандартний менеджер пакетів pip за допомогою однієї команди.
pip install pandas numpy pymavlink

Встановлення фронтенду
Клієнтська частина побудована на базі React 19 та збирача Vite. Для 3D-візуалізації використовується бібліотека React Three Fiber, а для елементів інтерфейсу підключені іконки Lucide React. Додатково налаштована система локалізації через i18next. Усі необхідні залежності вже прописані у конфігураційному файлі package.json. Щоб їх встановити, необхідно перейти у директорію фронтенду та запустити автоматичне завантаження пакетів.
cd react-app
npm install

Запуск проєкту
Коли всі складові налаштовані, можна переходити до запуску додатку. Для старту клієнтської частини відкрийте термінал у папці фронтенду та виконайте команду для запуску локального сервера розробки. Після цього додаток стане доступним у браузері за адресою, яку вкаже Vite.
npm run dev

Для запуску серверної частини необхідно відкрити окремий термінал безпосередньо у папці бекенд-проєкту та активувати його наведеною нижче командою. Обидва термінали мають працювати паралельно.
dotnet run

Теоретичне обґрунтування математичних перетворень
В основі обробки польотної телеметрії та 3D-візуалізації лежать спеціалізовані математичні алгоритми. У коментарях до сирцевого коду проєкту наведено детальне пояснення обраних підходів, які гарантують точність розрахунків та стабільність відображення об'єкта.

Для розрахунку та візуалізації орієнтації ракети у просторі використовуються кватерніони, а не традиційні кути Ейлера (крен, тангаж, рискання). Це рішення зумовлене необхідністю уникнення математичної аномалії, відомої як шарнірний замок (gimbal lock). При використанні кутів Ейлера, коли один з кутів досягає 90 градусів, осі обертання можуть збігтися, що призводить до втрати одного ступеня свободи і непередбачуваного обертання 3D-моделі. Кватерніони натомість використовують чотиривимірний вектор для опису тривимірних обертань, що гарантує безперервну, плавну та математично стабільну інтерполяцію рухів ракети під час будь-яких маневрів.

Також у коді враховано природу похибок при роботі з даними інерційного вимірювального модуля (IMU). Проєкт уникає прямого подвійного інтегрування показників акселерометра для визначення абсолютної позиції об'єкта. Оскільки сирі дані датчиків містять апаратний шум та зміщення, перше інтегрування (для отримання швидкості) накопичує лінійну помилку, а друге інтегрування (для отримання координат) призводить до квадратичного зростання похибки, що викликає стрімкий дрейф позиції. Тому логіка програми застосовує гібридний підхід: високочастотні дані IMU використовуються для розрахунку орієнтації та згладжування векторів швидкості, тоді як абсолютна позиція у просторі жорстко прив'язується до більш стабільних даних GPS для запобігання накопиченню математичної похибки.

Визначення точки посадки та побудова самої траєкторії вимагають переведення географічних координат у локальну метричну систему. Алгоритм фіксує подію посадки лише після проходження точки апогею (максимальної висоти), перевіряючи два критерії: падіння висоти нижче нульової відмітки старту (жорстке приземлення) або зниження вертикальної та горизонтальної швидкостей майже до нуля (м'яка посадка). Для точного відображення цього місця на 3D-сцені сирі координати GPS (широта та довгота) трансформуються у локальну декартову систему координат (X, Y у метрах) відносно стартового майданчика за допомогою формули гаверсинуса (Haversine formula). Це дозволяє нівелювати спотворення від кривизни Землі та отримати точну геометричну параболу польоту від моменту запалювання до точки приземлення.
